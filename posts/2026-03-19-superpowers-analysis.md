## 为什么 AI 编程助手会"翻车"

在没有任何约束的情况下，AI 编程助手通常会做这些事：

- 看到需求就立刻写代码，跳过所有设计讨论
- 用"这应该能工作"替代真正的验证
- 在调试时随机尝试几个修复，直到错误消失（而不是找到根因）
- 说"完成了"，但实际上测试还没跑

这些问题不是模型能力不足，而是**缺乏流程约束**。一个经验丰富的工程师之所以靠谱，不是因为他们更聪明，而是因为他们遵循了一套经过验证的工作流程。

[superpowers](https://github.com/obra/superpowers) 要做的正是这件事：给 AI 编程助手装上一套强制性的工程实践框架。

---

## 什么是 Skill

superpowers 把每一种工程实践封装成一个 **Skill**（技能文件）。Skill 不是提示词，而是一份详细的操作手册，里面包含：

- **何时使用**：精确的触发条件
- **核心流程**：分步骤的强制工作流
- **红线清单**：不允许做的事情和常见的绕过借口
- **验证标准**：怎么确认这一阶段已经真正完成

当 AI 遇到某类任务时，它必须先调用对应的 Skill，再按照 Skill 的规定行动。"如果有 1% 的可能性某个 Skill 适用，就必须调用它"——这是框架的核心规则。

目前 superpowers 共有 14 个 Skill，分为五类。

---

## 设计与规划类（3 个）

### brainstorming — 强制设计对话

这是所有工作流的起点。**在用户批准设计之前，禁止写任何一行代码**。

流程如下：

```
1. 探索项目上下文（已有代码、文档、最近提交）
2. 逐个提问，澄清需求（一次只问一个问题）
3. 提出 2-3 种设计方案，列出各自的取舍
4. 迭代修改，最多 3 轮评审
5. 将确认的设计写入 docs/superpowers/specs/ 目录
6. 交棒给 writing-plans
```

一个容易被忽视的洞察：**"简单项目"往往是假设最多、浪费最多的地方**。如果需求看起来很简单就直接跳过设计阶段，后期的返工成本可能远超前期讨论的成本。

### writing-plans — 把设计拆成两分钟的任务

设计通过后，writing-plans 负责把它翻译成可执行的实施计划。关键要求：

- **每个任务 2-5 分钟**，粒度足够小，进度可见
- 必须写明具体文件路径，不允许含糊的"在合适的地方"
- 每个任务都有完整代码示例和预期输出
- 遵循 TDD 节奏：写失败测试 → 运行确认失败 → 实现代码 → 确认通过 → 提交

计划本身也要经过专门的 reviewer subagent 审核，确认计划的完整性后才能进入执行阶段。

### using-git-worktrees — 隔离工作环境

在开始执行之前，先创建独立的 git worktree，让功能开发与主线完全隔离。创建后会自动运行安装依赖、执行基线测试，确认起点是干净的。

## 执行类（3 个）

### subagent-driven-development — 每个任务派一个新 Agent

这是效率提升最显著的设计。执行计划时，**每个任务都分发给一个全新的 subagent**，而不是在同一个上下文里连续执行。

为什么要用新的 subagent？

1. **避免上下文污染**：连续执行多个任务后，AI 对早期任务的记忆会干扰当前判断
2. **强制精确传递上下文**：主 Agent 必须明确地告诉 subagent 它需要知道的一切，这个过程本身就能暴露信息缺口
3. **并行安全**：不同任务的 subagent 互不干扰

每个任务完成后要经历两阶段审查：

```
第一阶段：Spec 合规性审查
  → 实现是否满足规格要求？有没有多做或少做？

第二阶段：代码质量审查
  → 代码质量是否可以接受？

两个阶段都通过 → 才能标记任务完成，进入下一个任务
```

审查也是由 subagent 独立完成的，这保证了审查者不会因为"自己写的代码"而有偏见。

根据任务复杂度，框架还会指导使用不同能力的模型：

| 任务类型 | 推荐模型 |
|--------|---------|
| 单文件、规格清晰的机械性任务 | 轻量/廉价模型 |
| 跨文件协调、集成类任务 | 标准模型 |
| 架构设计、代码评审 | 最强模型 |

### executing-plans — 跨 session 的批量执行

与 subagent-driven-development 不同，这个 Skill 用于在**单独 session 里**分阶段执行计划。每个阶段有明确的检查点，完成后等待用户确认再继续。

### finishing-a-development-branch — 受控的收尾

开发完成后，这个 Skill 负责结尾工作。它强制：

1. 先确认所有测试通过，才能进入下一步
2. 给出四个选项：本地合并 / 推送创建 PR / 保留分支 / 丢弃
3. 如果选择丢弃，要求用户明确输入 "discard" 确认，防止误操作

## 测试与验证类（2 个）

### test-driven-development — TDD 铁律

这是框架里最硬的约束之一。

> **在写生产代码之前，必须先有一个失败的测试。**

具体流程是经典的红-绿-重构：

```
RED：写一个最简单的、演示目标行为的测试
     ↓
验证 RED：运行测试，确认它以正确的原因失败
     ↓（如果测试意外通过了，说明测试本身有问题）
GREEN：写满足测试的最简单实现
     ↓
验证 GREEN：确认测试通过，已有测试全部通过
     ↓
REFACTOR：在绿灯状态下消除重复、改善命名
```

"如果你没有亲眼看到测试失败，你就不知道它到底在测试什么。"——验证失败这一步是强制的，不能跳过。

### verification-before-completion — 完成声明的门禁

> **在没有新鲜验证证据之前，不允许声称工作已完成。**

这个 Skill 针对的是 AI 的一个常见问题：在没有实际运行测试的情况下说"完成了"。

五步验证门禁：

1. 确定能证明断言的具体命令
2. 重新执行这个命令（不能用缓存结果）
3. 完整读取输出和退出码
4. 验证输出是否真的支持该声明
5. 只有在确认之后，才能做出"完成"的声明

禁止出现的表述："应该能工作"、"看起来没问题"、"大概是"——这些都是没有证据的声明。

## 调试类（2 个）

### systematic-debugging — 系统性调试

遇到 bug 时，禁止随机尝试修复。必须按四个阶段进行：

```
阶段一：根因调查
  → 读完整的错误信息
  → 稳定复现问题
  → 回溯最近的变更
  → 从症状反向追踪数据流

阶段二：模式分析
  → 找到相似的可运行代码
  → 完整对比两者的差异（而不是只看表面）
  → 记录每一个不同点

阶段三：假设验证
  → 用文字写下假设（强迫自己明确化）
  → 用最小的变更验证假设
  → 确认假设成立后再继续

阶段四：实现修复
  → 先写失败测试，复现 bug
  → 实现针对根因的单一修复
```

护栏：如果尝试了 3 次以上的修复都没有成功，必须停下来质疑架构本身，而不是继续打补丁。"随机修复浪费时间，还会制造新的 bug。"

### dispatching-parallel-agents — 并行调试多个独立问题

当测试套件里有多个互相独立的失败时，可以同时派遣多个 subagent 分别处理，而不是串行调试。

每个 subagent 的任务描述必须做到：

- **聚焦**：只负责一个问题域
- **自包含**：包含解决该问题所需的所有上下文
- **边界清晰**：明确说明哪些文件可以修改，哪些不能

## 代码评审类（2 个）

### requesting-code-review — 主动触发评审

完成任务后，主动把工作交给 code-reviewer subagent 评审。关键设计：reviewer 接收的是**专门构建的上下文**（原始需求 + 提交 SHA + 变更描述），而不是主 session 的历史。

这保证了评审者的客观性——它不会因为"看着代码写出来的过程"而对结果有先入为主的判断。

### receiving-code-review — 技术性回应，而不是社交性回应

收到评审意见时，必须做技术判断，而不是情绪反应。要避免的表述：

- "完全正确！我立刻修改。"（还没验证就同意）
- "很棒的建议！"（评审不是夸奖游戏）

正确流程：

1. 用自己的话复述需求，确认理解正确
2. 对照实际代码验证评审意见
3. 评估技术合理性（这个项目、这个上下文下）
4. 同意：附上技术理由；不同意：用证据提出异议
5. 每次只修改一个问题，修改后立即测试

## 框架类（2 个）

### using-superpowers — 一切的前提

这是元技能：告诉 AI 怎么使用整个 superpowers 系统。

核心规则：**有 1% 的可能性某个 Skill 适用，就必须先调用它，再做任何其他事情**。

Skill 里有一张"自我欺骗检测表"，专门针对 AI 绕过规则时的常见借口：

| 借口 | 真相 |
|-----|-----|
| "这只是个简单问题" | 问题也是任务，检查 Skill |
| "我需要先收集信息" | Skill 告诉你怎么收集信息 |
| "这个 Skill 太重了" | 简单的事情会变复杂，用 Skill |
| "我记得这个 Skill 的内容" | Skill 会更新，重新读取 |

### writing-skills — 用 TDD 写流程文档

可以创建自定义 Skill 扩展框架。有趣的是，写 Skill 本身也遵循 TDD：

```
RED：不给 AI 看 Skill，在目标场景下观察它的基准行为和偷懒借口
GREEN：针对这些借口写最小化的 Skill，再次运行场景
REFACTOR：堵住发现的新漏洞，重复测试直到无法绕过
```

这意味着 Skill 的正确性是可以测试和迭代的。

## 整体工作流：七个阶段

把 14 个 Skill 串联起来，完整的开发生命周期是这样的：

```
用户提需求
    │
    ▼
brainstorming ──────── 设计对话 + 方案确认
    │                  写入 docs/superpowers/specs/
    ▼
using-git-worktrees ── 创建隔离分支 + 基线测试
    │
    ▼
writing-plans ─────── 拆分任务 + 计划评审
    │                  写入 docs/superpowers/plans/
    ▼
subagent-driven-development
    │    ┌─ 分发 subagent 执行任务
    │    ├─ Spec 合规性评审
    │    ├─ 代码质量评审
    │    └─ 循环直到所有任务完成
    │
    ▼
finishing-a-development-branch
         验证测试 → 合并 / PR / 保留 / 丢弃
```

整个过程中，TDD 和 verification-before-completion 贯穿每个实现步骤。

## 为什么这样设计能提升 AI 质量

superpowers 的核心洞察是：**AI 的失误很少来自能力不足，更多来自缺乏约束**。

传统上，工程师之所以靠谱，是因为他们经过训练后内化了一套工作流程——写代码前先想清楚，测试通过才算完成，提交前要 review。但 AI 没有这个内化过程，它默认行为是"看到问题就回答"。

superpowers 做的事情是：

1. **把隐性的工程规范显性化** — 用 Skill 文件把"好的工程师会怎么做"写成可执行的步骤
2. **用强制流程替代自律** — 不依赖 AI 自觉遵守，而是让绕过变得困难
3. **用独立评审对抗确认偏误** — subagent reviewer 没有"这是我写的代码"的心理包袱
4. **用验证门关对抗乐观偏误** — 必须有证据才能声称"完成"

从架构角度看，这是把软件工程的**过程控制**（process control）移植到 AI 工作流的一次尝试。它的效果在于：不是让 AI 更聪明，而是让 AI 更系统、更可预期。

## 使用方式

```bash
# 在 Claude Code 中安装
/plugin install superpowers@claude-plugins-official

# 更新
/plugin update superpowers
```

安装后，Skill 会自动注册到 Claude Code 的工具列表中，在合适的时机被触发。

目前支持的平台：Claude Code、Cursor、Codex、OpenCode、Gemini CLI。

## 总结

superpowers 本质上是**一套软件工程方法论的 AI 实现**。它把 TDD、系统性调试、设计评审、验证门关这些工程实践，翻译成了 AI 可以强制执行的 Skill 文件。

对于想在生产项目里更稳定地使用 AI 编程助手的团队来说，这套框架的价值不在于每个 Skill 本身，而在于它们组合起来形成的**闭环工作流**：设计 → 规划 → 执行 → 验证 → 评审 → 收尾，每一步都有明确的输入、输出和验收标准。

这让 AI 从一个"很快但不可靠"的代码生成器，变成了一个"有流程、有验证、有记录"的工程协作者。
